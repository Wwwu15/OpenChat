# 聊天输入区重设计 Spec

日期：2026-06-18  
项目：Open Chat Android App  
范围：主聊天界面的底部聊天输入区（composer）与其相关状态展示

## 1. 背景

当前聊天输入区存在以下问题：

- `发送` 与 `停止` 为两个独立按钮，状态切换割裂
- 输入框更像单行短消息框，不适合提示词式长文本输入
- 附件入口、附件状态、发送状态分散，信息密度低且不直觉
- 流式生成时的状态反馈偏弱，用户不容易一眼判断当前阶段
- 图片、文档、长文本等不同发送形态没有被统一进同一个输入容器

本次目标是将聊天输入区重设计为更接近主流 AI 聊天产品的集成式 composer，同时保持现有 App 的整体视觉风格与页面结构不被大改。

## 2. 设计目标

本次重设计应满足以下目标：

1. 使用单一主按钮在 `发送 / 停止` 两种状态之间切换
2. 输入区支持多行与自动增高，更像“写提示词”而非“发短信”
3. 将图片、文档等待发内容统一显示在输入框上方
4. 明确区分“待发送附件”与“已发送消息中的附件”
5. 让生成中状态与主按钮、输入区形成统一反馈
6. 保持暗色模式、现有页面过渡动画、历史消息流、附件能力继续可用

## 3. 非目标

本次不包含以下内容：

- 不新增新的附件类型
- 不调整聊天消息气泡的整体布局规则
- 不重写历史消息中的附件展示逻辑
- 不修改 API 管理、Markdown 开关、暗色模式等其它页面的信息架构
- 不引入回车发送；回车仍然只用于换行

## 4. 核心交互总览

新的 composer 为一个完整的大圆角容器，内部纵向分为两层：

1. **待发送附件区**：位于容器上方
2. **输入与操作区**：位于容器下方

整体行为规则如下：

- 用户选择图片或文档后，不立即发送
- 所选附件先进入待发送附件区
- 用户输入文本后，可与待发附件一起发送
- 右下角只保留一个主按钮
- 主按钮在空闲态为 `发送`，在生成态切换为 `停止`

## 5. 界面结构

### 5.1 Composer 外层容器

- 固定在聊天页面底部
- 保持大圆角、轻阴影、与当前应用视觉一致
- 支持 `imePadding` 与 `navigationBarsPadding`
- 内容高度随内部输入区和待发附件区变化而增长

### 5.2 待发送附件区

待发送附件区位于 composer 顶部，采用横向卡片队列形式。

规则如下：

- 占据一整行，但内部为横向排列
- 多个附件时可左右滑动
- 附件卡片与输入区之间保留稳定间距
- 仅在存在待发附件时显示；无附件时整块区域不占位

### 5.3 输入与操作区

输入与操作区位于 composer 下半部分，包含：

- 多行输入框
- 位于输入框左下角内侧的附件按钮（即当前 `+` 入口）
- 位于右下角的单一主按钮
- 生成中状态提示文本

## 6. 待发送附件区详细设计

### 6.1 总体形式

待发送附件区采用“横向滑动的大圆角卡片队列”。

每个附件卡片均包含：

- 左侧预览区
- 右侧文本信息区
- 右上角浮动删除按钮

卡片整体看起来应像“待发送内容条”，而不是聊天消息气泡。

### 6.2 图片附件卡片

图片附件卡片规则如下：

- 左侧显示图片缩略图
- 缩略图为圆角矩形
- 右上角叠加一个圆形 `X` 删除按钮
- 不展示图片文件名
- 如仅有图片且无文本，仍允许直接发送

### 6.3 文档附件卡片

文档附件卡片规则如下：

- 左侧显示文件类型图标，例如 Word / PDF / 通用文档图标
- 右侧第一行显示文件名
- 文件名过长时单行省略
- 右侧第二行显示状态文案
- 状态文案优先使用以下枚举：
  - `解析中...`
  - `已就绪`
  - `上传失败`
- 卡片右上角提供圆形 `X` 删除按钮

### 6.4 多附件排列

当存在多个待发附件时：

- 附件卡片横向并排
- 容器支持横向滚动
- 卡片间距固定
- 不换行堆叠，避免把 composer 高度迅速撑大

## 7. 输入区详细设计

### 7.1 输入框

输入框改为真正的多行输入体验：

- 默认显示占位文案
- 支持多行文本输入
- 回车仅换行，不触发发送
- 输入内容增加时自动增高
- 超过设定高度后输入框内部滚动

### 7.2 高度策略

建议的交互策略：

- 初始高度为紧凑单段高度
- 随文本增加逐步扩展
- 达到上限后不再继续推高整个 composer
- 超出部分在文本区内部滚动

目标是让长文本输入更自然，同时避免键盘弹出后把消息区域压缩得过于明显。

### 7.3 附件按钮

附件按钮定义如下：

- 即当前的 `+` 入口
- 放置在输入框左下角内侧
- 点击后打开现有附件选择逻辑
- 继续复用当前图片 / 文件选择能力

它的职责仅为“添加待发送附件”，不承担发送动作。

## 8. 主按钮设计

### 8.1 单一主按钮

右下角只保留一个圆形主按钮。

该按钮根据状态切换：

- 空闲可发送时：显示发送图标
- 正在生成时：切换为停止图标

### 8.2 可用性规则

按钮状态规则如下：

- 无文本且无附件：按钮弱化，不可触发发送
- 有文本或有附件：按钮高亮，可发送
- 生成中：按钮切换为可点击的停止按钮

### 8.3 停止行为

点击停止按钮后：

- 取消当前流式生成任务
- 保留已生成的部分助手内容
- 主按钮恢复到发送态

不再额外显示第二个停止按钮。

## 9. 状态设计

### 9.1 状态 A：发送前（空白）

- 不显示待发送附件区
- 输入框为空
- 主按钮弱化
- 附件按钮可点击

### 9.2 状态 B：发送前（可发送）

满足以下任一条件时进入可发送态：

- 输入框有非空文本
- 存在至少一个待发附件

表现为：

- 主按钮高亮
- 输入区允许继续编辑

### 9.3 状态 C：发送图片

- 图片显示在待发送附件区
- 主按钮进入可发送态
- 用户可补充文本，也可直接发送

### 9.4 状态 D：发送文档

- 文档以文件卡片形式显示在待发送附件区
- 显示文件名和状态文案
- 主按钮在文档准备就绪后保持可发送

### 9.5 状态 E：发送长文本

- 输入框高度自动增高
- 达上限后内部滚动
- 主按钮保持固定位置，不跟随文本高度失控位移

### 9.6 状态 F：生成中

- 主按钮由发送切换为停止
- 输入区显示明确状态提示，如 `正在生成...`
- 待发送附件区清空
- 用户仍能看到输入区，但当前发送过程已开始

## 10. 生成中反馈

为了增强流式输出反馈，生成中状态应同时通过两处表达：

1. **主按钮图标切换**：发送 -> 停止
2. **输入区状态文案**：显示 `正在生成...`

状态文案应属于 composer 内部的一部分，而不是远离输入区单独悬浮，避免用户视线来回寻找状态。

## 11. 数据与状态流

### 11.1 待发送态

本地 UI 状态需显式区分：

- 当前输入文本
- 待发送附件列表
- 当前是否处于生成中
- 当前是否存在错误

### 11.2 发送动作

点击发送时：

1. 读取当前文本与待发送附件
2. 组装请求内容
3. 将待发送内容加入当前会话消息流
4. 清空 composer 内的待发送附件区
5. 主按钮切换到停止态
6. 开始流式接收

### 11.3 停止动作

点击停止时：

1. 取消当前请求 Job
2. 保留已收到的 assistant 文本
3. 更新 `isSending` 状态
4. 主按钮恢复发送态

## 12. 错误处理

### 12.1 附件阶段错误

待发送附件阶段的错误应尽量就地表达：

- 文档解析失败：在文档卡片状态位显示失败信息，或给出轻量提示
- 不支持的附件类型：沿用现有错误提示机制

### 12.2 发送阶段错误

请求失败时：

- 停止生成状态
- 主按钮恢复发送态
- 保留用户原输入与待发附件，便于重试
- 沿用现有错误卡片 / 错误提示逻辑

## 13. 与现有实现的衔接要求

本次重设计需兼容以下既有能力：

- 暗色模式与非暗色模式
- 现有 API 配置和会话逻辑
- 图片与文档附件能力
- 流式输出中断能力
- 主聊天页已有动画与导航逻辑

同时应特别注意：

- 新 composer 不得遮挡底部系统区域
- 键盘弹出时消息列表仍应优先从顶部自然展示
- 长文本与多附件组合时，不应造成底部控件重叠

## 14. 实现边界建议

建议将本次改动尽量收敛在以下区域：

- `AiComposer.kt`：重做 composer UI 结构
- `ChatScreen.kt`：调整待发送附件区与发送中状态展示
- `ChatViewModel.kt`：细化发送 / 停止 / 附件待发状态管理

如需新增小型 UI 子组件，可拆分但应保持职责单一，例如：

- `PendingAttachmentStrip`
- `PendingImageCard`
- `PendingDocumentCard`
- `ComposerPrimaryActionButton`

## 15. 验收标准

本次设计完成后，应满足以下验收标准：

1. 底部只存在一个主操作按钮，且能在发送 / 停止间切换
2. 输入框支持多行、自动增高、回车换行
3. 附件按钮位于输入框左下角内侧
4. 待发图片以缩略图卡片显示在输入框上方
5. 待发文档以横向文件卡片显示在输入框上方，并显示文件名
6. 多附件可横向滑动浏览
7. 生成中状态有明确反馈，且与主按钮联动
8. 暗色模式和非暗色模式下配色与层次保持一致
9. 键盘弹出时 composer 不出现重叠、错位、异常留白
10. 现有图片 / 文档发送能力和停止生成功能继续可用

## 16. 推荐实现方向

推荐采用“集成式 AI Composer”方案：

- 一个整体大圆角容器
- 顶部横向待发附件卡片区
- 中部多行输入区
- 左下角内置附件按钮
- 右下角单一发送 / 停止主按钮
- 底部或输入区内显示生成中状态

这是当前最符合用户给定参考图、也最适合现有 App 视觉延续性的方案。
